#### 신호

##### 트레이스

**트레이스**는 애플리케이션으로 요청이 만들어질 때 어떤 일이 일어나는지에 대한 큰 그림을 제공한다. 애플리케이션이 하나의 데이터베이스를 가진 모놀리스이든, 복잡한 서비스의 메시(mesh)이든, 트레이스는 애플리케이션에서 요청의 전체 "경로"를 이해하는 데 필수적이다.

이를 [스팬][otel-concepts-traces-spans]으로 표현된 세 개의 작업 단위로 알아보자:

> [!NOTE]
>
> 다음 JSON 예시들은 특정한 형식을 나타내지 않으며 특히 더 장환한 [OTLP/JSON][otel-specs-otlp-json-protobuf-encoding]은 아니다.

`hello` 스팬:

```json
{
  "name": "hello",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "051581bf3cb55c13"
  },
  "parent_id": null,
  "start_time": "2022-04-29T18:52:58.114201Z",
  "end_time": "2022-04-29T18:52:58.114687Z",
  "attributes": {
    "http.route": "some_route1"
  },
  "events": [
    {
      "name": "Guten Tag!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

이것이 전체 연산의 시작과 끝을 나타내는 루트 스팬이다. 트레이스를 나타내는 `trace_id` 필드를 갖지만 `parent_id`를 갖지 않는다. 이것으로 루트 스팬임을 알 수 있다.

`hello-greetings` 스팬:

```json
{
  "name": "hello-greetings",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "5fb397be34d26b51"
  },
  "parent_id": "051581bf3cb55c13",
  "start_time": "2022-04-29T18:52:58.114304Z",
  "end_time": "2022-04-29T22:52:58.114561Z",
  "attributes": {
    "http.route": "some_route2"
  },
  "events": [
    {
      "name": "hey there!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    },
    {
      "name": "bye now!",
      "timestamp": "2022-04-29T18:52:58.114585Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

이것은 인사(greetings)와 같은 특정 작업을 캡슐화하고 부모가 `hello`다. 루트 스팬과  같은 `trace_id`를 공유하는 것은 같은 트레이스의 일부임을 나타낸다. 추가로, `hello` 스팬의 `span_id`와 일치하는 `parent_id`을 갖는다.

`hello-sallutations` 스팬:

```json
{
  "name": "hello-salutations",
  "context": {
    "trace_id": "5b8aa5a2d2c872e8321cf37308d69df2",
    "span_id": "93564f51e1abe1c2"
  },
  "parent_id": "051581bf3cb55c13",
  "start_time": "2022-04-29T18:52:58.114492Z",
  "end_time": "2022-04-29T18:52:58.114631Z",
  "attributes": {
    "http.route": "some_route3"
  },
  "events": [
    {
      "name": "hey there!",
      "timestamp": "2022-04-29T18:52:58.114561Z",
      "attributes": {
        "event_attributes": 1
      }
    }
  ]
}
```

이 스팬은 이 트레이스에서 세 번째 연산을 나타내며 이전 것과 같이 `hello` 스팬의 자식이다. 즉, `hello-greetings` 스팬의 형제다.

이 세 블록의 JSON은 모두 같은 `trace_id`를 공유하고 `parent_id` 필드는 계층을 나타낸다. 이렇게 트레이스가 만들어진지다!

하나 더 알아둬야 할 것은 각 스팬은 구조화된 로그처럼 보인다는 것이다. 이는 일종의 로그이기 때문이다! 트레이스를 생각하는 한 가지 방법은 컨텍스트, 관계(correlation), 계층 구조 등을 가진 구조화된 로그의 모음으로 보는 것이다. 하지만 이러한 "구조화된 로그"는 서로 다른 프로세스, 서비스, VMs, 데이터 센터 등으로부터 올 수 있다. 이것이 트레이싱이 어떤 시스템에서든 엔드 투 엔드 뷰를 나타낼 수 있게 하는 요소다.

OpenTelemetry에서 트레이싱이 어떻게 동작하는지 이해하기 위해 코드를 계측할 때 동작하는 구성 요소 목록을 살펴보자.
