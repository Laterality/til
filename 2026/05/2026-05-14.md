###### 계측 대상

**중첩된 네트워크와 다른 스팬**

네트워크 호출은 주로 해당하는 클라이언트 구현체를 통해 OpenTelemetry 자동 게측으로 추적된다.

만약 OpenTelemetry가 네트워크 클라이언트 추적을 지원하지 않으면, 취할 행동을 결정하는데 도움을 줄 만한 고려사항들이 있다:

- 네트워크 호출을 추적하는 것이 사용자의 관측가능성이나 이들을 지원하는 여러분의 능력을 향상시키는가?
- 라이브러리가 공개되고 문서화된 RPC  API에 기반한 래퍼인가? 사용자가 문제가 발생했을 때 기반 서비스로부터 지원을 받아야 하는가?
  - 라이브러리를 계측하고 개발 네트워크 시도를 추적하라.
- 이러한 호출들을 스팬으로 추적하는 것이 매우 장황한가? 혹은 성능에 상당한 영향을 주는가?
  - 로그 레벨(verbosity)과 함께 로그를 사용하거나 스팬 이벤트를 사용하라. 로그는 부모(공개 API 호출)과 연관될 수 있지만, 스팬 이벤트는 공개 API 스팬에 설정돼야 한다.
  - 이것들이 (유니크 트레이스 컨텍스트를 운반하고 전파하기 위해)스팬이어야 한다면 이들을 구성 옵션에 넣고 기본적으로 비활성화하라.

만약 OpenTelemetry가 이미 여러분의 네트워크 호출 추적을 지원한다면, 이를 중복해서 하고 싶지 않을 것이다. 몇 가지 예외가 있다:

- 자동 게측이 없는 사용자를 지원하기 위해, 특정 환경에서 혹은 사용자가 몽키 패칭을 걱정할 때 동작하지 않을 수 있다.
- 커스텀 혹은 레거시 연관관계와 컨텍스트 전파 프로토콜을 기반 서비스와 함께 활성화하기 위해.
- RPC 스팬을 필수 라이브러리나 자동 게측으로 커버되지 않는 서비스 특화 정보로 보충할 때.

중복을 피하기 위한 일반적인 솔루션은 아직 만드는 중이다.
