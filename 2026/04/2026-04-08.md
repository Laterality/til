###### 구조화, 비구조화, 반구조화(semistructured)된 로그

**구조화된 로그**

구조화된 로그는 다운스트림 시스템이 안정적으로 파싱하고 해석할 수 있는 일관된 스키마가 정의되거나 타입이 정해진 필드가 있는 로그다. 텍스트 인코딩은 JSON, protobuf 또는 또다른 형식이 될 수 있지만 로그를 구조화되게 만드는 것은 그저 유효한 JSON이 아니라 안정적인 스키마다. 예를 들어, 구조화된 JSON 로그는 다음과 같은 형태이다:

```json
{
  "timestamp": "2024-08-04T12:34:56.789Z",
  "level": "INFO",
  "service": "user-authentication",
  "environment": "production",
  "message": "User login successful",
  "context": {
    "userId": "12345",
    "username": "johndoe",
    "ipAddress": "192.168.1.1",
    "userAgent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/104.0.0.0 Safari/537.36"
  },
  "transactionId": "abcd-efgh-ijkl-mnop",
  "duration": 200,
  "request": {
    "method": "POST",
    "url": "/api/v1/login",
    "headers": {
      "Content-Type": "application/json",
      "Accept": "application/json"
    },
    "body": {
      "username": "johndoe",
      "password": "******"
    }
  },
  "response": {
    "statusCode": 200,
    "body": {
      "success": true,
      "token": "jwt-token-here"
    }
  }
}
```

인프라스트럭쳐 구성요소의 경우, 공용 로그 형식(CLF, Common Log Format)이 흔히 쓰인다:

```
127.0.0.1 - johndoe [04/Aug/2024:12:34:56 -0400] "POST /api/v1/login HTTP/1.1" 200 1234
```

또한 하이브리드 혹은 확장된 형식(예를 들어 JSON 블롭을 끝에 조합한 CLF 필드처럼)도 흔하다.

```
192.168.1.1 - johndoe [04/Aug/2024:12:34:56 -0400] "POST /api/v1/login HTTP/1.1" 200 1234 "http://example.com" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/104.0.0.0 Safari/537.36" {"transactionId": "abcd-efgh-ijkl-mnop", "responseTime": 150, "requestBody": {"username": "johndoe"}, "responseHeaders": {"Content-Type": "application/json"}}
```

이 경우 다운스트림 도구가 일관되게 분석할 수 있는 정규화된 레코드로 파싱하거나 추출한다. [OpenTelemetry 컬렉터][otel-collector]의 `filelogreceiver`는 혼합된 형식들을 파싱하는 헬퍼를 제공한다.

안정적인 스키마가 유효성을 검증하고 파싱하고 트레이스 및 메트릭과 연관시키고 대규모로 분석하기에 직관적으로 만들기 때문에 구조화된 로그는 프로덕션에서 선호된다.

[otel-collector]: https://opentelemetry.io/docs/collector/
