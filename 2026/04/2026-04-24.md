## OpenTelemtry

### 개념

#### 신호

##### 프로파일

##### 프로파일이 어떻게 다른 신호들을 보완하는가

각 OpenTelemetry 신호들은 다른 질문을 한다:

| 신호         | 질문                                                               |
| ------------ | ------------------------------------------------------------------ |
| **로그**     | 어떤 이산(discrete) 이벤트가 발생했는가? (시스템 동작에 대한 통찰) |
| **메트릭**   | 시스템 수준에 무슨 일이 벌어지고 있는가? (e.g. CPU 사용량이 90%)   |
| **트레이스** | 요청이 분산된 시스템을 따라 어떻게 이동하는가?                     |
| **프로파일** | 어느 코드가 리소스 소비의 책임을 갖고 있는가?                      |

OpenTelemetry 프로파일은 다른 신호들들의 양방향 링크를 지원한다. 이 상관관계는 두 차원을 따라 동작한다:

- **요청 컨텍스트 상관관계**: 프로파일링 데이터를 특정 트레이스나 스팬과 링크해 특정 요청 기간동안 무슨 코드가 실행됐는지 볼 수 있다.
- **리소스 컨텍스트 상관관계**: 프로파일링 데이터를 같은 서비스 인스턴스처럼 동일한 메트릭, 로그, 트레이스을 방출한 [리소스][otel-concepts-resources]에 링크한다.

프로파일은 다른 신호들과 연관시켰을 때 특히 강력해진다:

- **로그와 프로파일**: 메모리 부족 로그 엔트리로부터 메모리 압력에 책임을 갖는 코드 경로를 찾는다.
- **메트릭과 프로파일**: CPU나 메모리 사용량 스파이크에서 해당 리소스를 사용하는 함수를 바로 찾는다.
- **트레이스와 프로파일**: 트레이스에서 늘니 스팬으로부터 해당하는 프로파일을 보고 지연에 책임이 있는 코드를 식별한다.

[otel-concepts-resources]: https://opentelemetry.io/docs/concepts/resources/
